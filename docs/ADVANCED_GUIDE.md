# Advanced Guide

This document explains how the deployer works end-to-end for advanced users who want to understand or modify behavior.

[!NOTE]
This document was generated by AI, so if you see any inaccuracies or have suggestions for improvement, please submit a PR or open an issue.

## 1. High-level architecture

The deploy process has three layers:

1. Host launcher script
- `scripts/run.sh` (macOS/Linux) or `scripts/run.ps1` (Windows)
- Loads `.env` defaults
- Prompts for required runtime values
- Builds and runs the deployer Docker image

2. Deployer container entrypoint
- `scripts/container-run.sh`
- Validates required input values
- Generates a temporary Ansible inventory
- Runs the Ansible playbook with mode-specific variables

3. Remote provisioning and application deployment
- `ansible/playbook.yml`
- Installs system dependencies on Raspberry Pi
- Installs Docker Engine + Compose plugin + Nginx
- Renders and applies `docker-compose.yml` for MeshMonitor
- Applies firewall allow rules for port 80 when UFW or firewalld is active

## 2. Runtime flow in detail

### Step A: collect input

Required inputs for the main flow:
- Pi IP (`TARGET_PI_IP`)
- Pi password (`TARGET_PI_PASSWORD`)
- Radio mode (`RADIO_CONNECTION_TYPE` = `wifi` or `bluetooth`)

Mode-specific inputs:
- `wifi`: requires `RADIO_IP`
- `bluetooth`: requires `RADIO_MAC`

The launchers pass these as environment variables into the deployer container.

### Step B: entrypoint validation

`container-run.sh` enforces:
- Pi IP present
- Pi password present
- Radio mode is valid (`wifi` or `bluetooth`)
- If wifi mode: `RADIO_IP` must be valid IPv4
- If bluetooth mode: `RADIO_MAC` must match `AA:BB:CC:DD:EE:FF`

If validation fails, the script exits before Ansible runs.

### Step C: temporary inventory generation

The entrypoint creates `/workspace/ansible/inventory.ini` with a single host:
- Group: `[raspberry_pi]`
- Host alias: `pi_target`
- Connection: SSH with password auth
- Sudo (`become`) password is set to the same Pi password

### Step D: playbook execution

The entrypoint runs:
- `ansible-playbook /workspace/ansible/playbook.yml`
- Extra vars include mode and radio endpoint values

The playbook then:
- Bootstraps Python 3 if missing
- Validates mode and mode-specific radio values again
- Installs core packages and Docker repository
- Installs Docker + Compose plugin
- Installs Nginx
- In bluetooth mode, installs bluetooth dependencies (`bluez`, `bluez-tools`, `dbus`)
- Templates compose and Nginx config files
- Starts containers with `docker compose up -d`
- Reloads/restarts Nginx
- Applies firewall allow for HTTP if applicable

## 3. Compose rendering behavior by mode

Template source:
- `ansible/templates/docker-compose.meshmonitor.yml.j2`

### Wifi mode

Only `meshmonitor` service is deployed.

Environment variables include wifi radio endpoint values, such as:
- `MESHMONITOR_RADIO_TRANSPORT=wifi`
- `MESHMONITOR_RADIO_IP=<RADIO_IP>`

### Bluetooth mode

Two services are deployed:
- `meshtastic-ble-bridge`
- `meshmonitor`

`meshmonitor` depends on `meshtastic-ble-bridge` and is configured to use it via service DNS on the compose network.

## 4. Network exposure model

- MeshMonitor container port 3001 is published to Pi host port `MESHMONITOR_HTTP_PORT` (default `8080`).
- Nginx listens on host port 80 and reverse proxies to `127.0.0.1:<MESHMONITOR_HTTP_PORT>`.
- Users access the UI at `http://<PI_IP>/`.

Firewall handling in the playbook:
- UFW: if installed and active, runs `ufw allow 80/tcp`
- firewalld: if installed and running, enables `http` service

## 5. Configuration model

Common variables:
- `TARGET_PI_IP`
- `TARGET_PI_PASSWORD`
- `PI_USERNAME` (default `pi`)
- `PI_SSH_PORT` (default `22`)
- `RADIO_CONNECTION_TYPE`

Mode variables:
- Wifi: `RADIO_IP`
- Bluetooth: `RADIO_MAC`

Optional image and port overrides:
- `DEPLOYER_IMAGE_NAME`
- `MESHMONITOR_IMAGE`
- `MESHMONITOR_HTTP_PORT`
- `MESHTASTIC_BLE_BRIDGE_IMAGE`
- `MESHTASTIC_BLE_BRIDGE_PORT`

## 6. Security notes

- SSH password auth is used by design for onboarding simplicity.
- Pi password is passed into container environment at runtime.
- For production hardening, consider:
  - Migrating to SSH key auth
  - Restricting network access to Pi port 80
  - Using HTTPS/TLS in front of Nginx
  - Rotating default credentials in upstream apps immediately

## 7. Typical failure points

1. SSH connection failures
- Wrong Pi IP
- SSH disabled on Pi
- Wrong username/password
- Pi not reachable from host machine

2. Radio endpoint issues
- Wifi mode: radio not actually on same network / wrong `RADIO_IP`
- Bluetooth mode: radio not paired/trusted on Pi / wrong `RADIO_MAC`

3. Docker package installation issues on Pi
- Network/DNS issues on Pi
- Unsupported distro release for Docker apt repo

4. Firewall/network segmentation
- Port 80 blocked by local network policy
- Client not on routable network to Pi

## 8. Extending this deployer

Common extension points:
- Add tags and role structure in `ansible/playbook.yml`
- Add additional compose services in `docker-compose.meshmonitor.yml.j2`
- Add preflight checks in `container-run.sh` before playbook execution
- Add non-interactive CI mode by relying fully on `.env` and skipping prompts in launch scripts

## 9. Quick command references

Build deployer image:

```bash
docker build -t meshmonitor-deployer:latest .
```

Run deployer image directly (advanced, bypass launcher scripts):

```bash
docker run --rm \
  -e TARGET_PI_IP=<PI_IP> \
  -e TARGET_PI_PASSWORD=<PI_PASSWORD> \
  -e RADIO_CONNECTION_TYPE=wifi \
  -e RADIO_IP=<RADIO_IP> \
  meshmonitor-deployer:latest
```

Bluetooth variant:

```bash
docker run --rm \
  -e TARGET_PI_IP=<PI_IP> \
  -e TARGET_PI_PASSWORD=<PI_PASSWORD> \
  -e RADIO_CONNECTION_TYPE=bluetooth \
  -e RADIO_MAC=AA:BB:CC:DD:EE:FF \
  meshmonitor-deployer:latest
```
